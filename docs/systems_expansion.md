# Packed-CKKS systems expansion

This experiment adds provider-side scaling, session lifecycle, exact key
material sizes, memory measurements, and real socket transport to the existing
single-request CKKS study.  It does not change the retrieval protocol or make
an access-pattern-privacy claim.

The primary result is
`results/system_revision/systems_expansion/windows_full.json`.  It contains all
per-request samples, all three restart measurements, process attestations, and
the environment snapshot.  The compact derived table is
`results/system_revision/systems_expansion/windows_scaling_summary.csv`; it is
not an independent measurement.

## Reproduction command

Run from `revision_workspace` with the fixed Python 3.11 environment.  The
thread variables are checked by the harness and execution is refused unless
they are all one (unless the explicitly non-comparable
`--allow-unfixed-threads` flag is supplied).

```powershell
$env:OMP_NUM_THREADS='1'
$env:MKL_NUM_THREADS='1'
$env:OPENBLAS_NUM_THREADS='1'
$env:NUMEXPR_NUM_THREADS='1'
$env:VECLIB_MAXIMUM_THREADS='1'
$env:BLIS_NUM_THREADS='1'

python `
  -m system.systems_expansion `
  --projected-docs-path results\system_revision\cache\E_docs_e5base_proj672_1000000.npy `
  --output results\system_revision\systems_expansion\windows_full.json `
  --dimension 672 --candidates 100 `
  --concurrency 1 2 4 8 16 `
  --scaling-restarts 3 --requests-per-worker 20 `
  --warmup-requests-per-worker 3 `
  --naive-concurrency 1 8 --naive-restarts 2 `
  --naive-requests-per-worker 4 --naive-warmup-requests-per-worker 1 `
  --session-restarts 5 --session-warm-requests 20 `
  --socket-warmups 3 --socket-repeats 30 `
  --request-pool-size 64 --seed 20260711
```

The complete run took 318.9 seconds.  It was executed in an isolated CPU
window: other CPU-heavy experiments were paused before the process started and
resumed only after it exited.  The level order was ascending in restarts zero
and two and descending in restart one.

The exact projected-document input was a 2,688,000,128-byte float32 NPY array
with shape `[1,000,000, 672]` and SHA-256
`68884b1e5e5ab4bb0b60dcda0d956793f1d1fe1d586c62e33392921890aab17e`.
The 64 normalized queries and candidate lists were generated by
`numpy.random.default_rng(20260711)`.

## Host and scope

The host was Windows 11 build 26200, Intel Core i5-14400F (10 physical cores,
16 logical CPUs), 32 GiB RAM, Python 3.11.15, NumPy 2.4.3, TenSEAL 0.3.16, and
Microsoft SEAL through `tenseal.sealapi`.  The loaded OpenBLAS pool reported one
thread.  Process affinity and priority were not changed.  The RTX 5060 was
present but CKKS did not use it.

The scaling timer intentionally isolates the provider: each closed-loop parent
thread sends pre-encrypted queries to one independently spawned public-only
worker, waits for its encrypted response, and immediately sends the next
request.  Client encryption and decryption are excluded from this saturation
timer and are measured separately in the session experiment.  Worker startup
and warmups are also excluded.  Every response selected for the numerical
audit was decrypted only after the measured workload.

## Provider scaling

The packed method completed 1,860 measured requests without a failure.  QPS is
the median of the three per-restart throughput values; latency percentiles are
pooled across the measured requests.

| Workers | QPS | p50 (ms) | p95 (ms) | p99 (ms) | Busy cores | Peak summed worker RSS |
|---:|---:|---:|---:|---:|---:|---:|
| 1 | 4.255 | 213.23 | 326.57 | 358.39 | 0.82 | 84.2 MiB |
| 2 | 8.273 | 218.55 | 347.07 | 464.38 | 1.61 | 168.8 MiB |
| 4 | 14.325 | 258.22 | 378.30 | 414.36 | 3.39 | 338.7 MiB |
| 8 | 22.124 | 347.36 | 462.91 | 545.16 | 6.81 | 677.3 MiB |
| 16 | 25.993 | 568.23 | 727.43 | 846.09 | 11.87 | 1,356.7 MiB |

Throughput therefore rose by 6.11 times from one to sixteen workers.  The
curve begins to saturate after eight workers: sixteen workers equal the host's
logical CPU count but exceed its ten physical cores.  This is a saturation
result, not evidence of linear scaling.

Peak per-worker RSS was 84--87 MiB for the packed workers and the sampled sum
grew almost linearly.  Summed RSS can double-count shared DLL and memory-mapped
pages, so it must not be described as unique physical memory.

The largest audited packed-score absolute error was
`7.2725e-6`.  Every worker attested that it reconstructed a public context,
had no secret key, exposed no decryptor attribute, and opened the projected
documents read-only.

## Packed versus naive at concurrency

The naive method was measured at one and eight workers and completed all 72
requests.  Its returned frame contains 100 ciphertexts, so these measurements
include the cost of transferring the approximately 13.1 MB response through
the local process pipe.

| Workers | Method | Median QPS | Pooled p50 (ms) | Pooled p95 (ms) | Pooled p99 (ms) |
|---:|---|---:|---:|---:|---:|
| 1 | one-response packed | 4.255 | 213.23 | 326.57 | 358.39 |
| 1 | naive per candidate | 0.426 | 2,081.02 | 3,998.63 | 4,180.49 |
| 8 | one-response packed | 22.124 | 347.36 | 462.91 | 545.16 |
| 8 | naive per candidate | 2.460 | 2,869.92 | 3,856.41 | 4,203.62 |

Packed throughput was 10.00 times higher at one worker and 8.99 times higher
at eight workers.  Its pooled median round trip was respectively 9.76 and 8.26
times shorter.  These ratios describe the complete serialized local IPC path,
not only the HE arithmetic kernel.

## Fresh and persistent sessions

Five independent sessions were created.  A fresh session means new key
material, a newly serialized public context, a new spawned server process, and
the first full encryption--request--decryption path.  The OS page cache was not
flushed, so “fresh” must not be renamed “cold disk cache.”

| Quantity | p50 (ms) | p95 (ms) | p99 (ms) |
|---|---:|---:|---:|
| Key/context setup and local public-context verification | 289.32 | 473.58 | 506.89 |
| Spawned server startup and context load | 375.59 | 392.85 | 394.65 |
| First full client request | 255.74 | 595.90 | 650.41 |
| Ready-to-first-result total | 928.95 | 1,324.08 | 1,342.17 |
| Persistent full client request | 254.85 | 527.97 | 582.91 |
| Persistent client encryption | 10.87 | 53.90 | 129.25 |
| Persistent process-pipe round trip | 232.96 | 510.24 | 557.69 |
| Persistent client decrypt/decode | 7.47 | 12.51 | 40.18 |

One persistent request reached 1.92 seconds; it is retained in the JSON rather
than trimmed.  TenSEAL's exposed SEAL serialization path uses temporary files,
so the measured tails include local filesystem and scheduler effects.

## Exact serialized sizes and amortization

The exact sizes below come from `SEAL::save` output and the application's
binary envelopes for one of the five fresh sessions.  Key compression makes a
few bytes vary across independently generated key sets, which is why the
values should be reported as a measured instance rather than a protocol
constant.

| Component | Bytes |
|---|---:|
| Encryption parameters | 80 |
| Public key | 366,322 |
| Ten power-of-two Galois-key objects | 7,326,779 |
| Public-context envelope overhead | 399 |
| Complete public server context | 7,693,580 |
| Client-only serialized secret key (never sent) | 181,290 |
| Encrypted query | 235,534 |
| 100 uint64 candidate IDs | 800 |
| Complete request application payload | 236,334 |
| One-ciphertext packed response | 131,876 |

No relinearization keys are generated or transmitted.  The server context has
no secret-key payload.

Including one public-context transfer, the effective application bytes per
request are 8,061,790 for a one-request session, 1,137,568 when amortized over
ten requests, and 445,145.8 when amortized over 100 requests.  The median
setup-time contribution similarly falls from 673.21 to 67.32 and 6.73 ms per
request.

## Real loopback TCP and TLS

The socket stage uses actual persistent IPv4 sockets bound to `127.0.0.1`,
`TCP_NODELAY`, deterministic length framing, and the same serialized CKKS
query/response envelopes.  TLS was negotiated as TLS 1.3 with
`TLS_AES_256_GCM_SHA384`. The archived run used one fixed self-signed
certificate, and client verification was disabled because the experiment
measures encryption and framing rather than production PKI. The released code
generates equivalent disposable loopback material in a temporary directory so
that no private key is stored in the repository.

| Transport | Round-trip p50 (ms) | p95 (ms) | p99 (ms) | Transport/framing p50 (ms) |
|---|---:|---:|---:|---:|
| TCP loopback | 212.36 | 239.42 | 242.37 | 0.498 |
| TLS 1.3 loopback | 217.77 | 240.87 | 255.26 | 0.979 |

The median TLS transport/framing increment over plain TCP was 0.481 ms.  The
application frames averaged 236,325.5 request bytes and 131,992 response bytes;
these counts include this experiment's socket frame but do not include TCP/IP
headers or TLS record headers because no packet capture was used.

This result must always be labelled **loopback**.  It establishes that the
serialized protocol works over real TCP/TLS and measures local framing cost;
it does not substitute for a controlled LAN/WAN experiment.

The focused transport stage can be rerun without the scaling/session stages:

```powershell
$py `
  -m system.systems_expansion `
  --loopback-only `
  --projected-docs-path results\system_revision\cache\E_docs_e5base_proj672_1000000.npy `
  --output results\system_revision\systems_expansion\loopback_only.json `
  --socket-warmups 3 --socket-repeats 30
```

## Same-CPU Windows/WSL portability point

An identical 20-repeat microbenchmark was run back-to-back on the same
i5-14400F in native Windows and Ubuntu 22.04 under WSL2.  Both executions used
the same source, seed `20260710`, TenSEAL 0.3.16/Microsoft SEAL, d=672, K=100,
N=8192, `[60,40,60]`, scale 2^40, two warmups, and one native thread.

This is not exact language-runtime parity.  Windows used Python 3.11.15 and
NumPy 2.4.3.  The Ubuntu security image exposed Python 3.10.12; the newest
compatible binary NumPy wheel was 2.2.6.  TenSEAL was the same version in both
cases.  The Linux packages were installed only with binary wheels into a
disposable `pip --target` directory under `revision_workspace/tmp` on D.  No
`apt`, Sage, base-distribution, or user-site state was changed.

| Runtime | Kernel | Server p50 (ms) | p95 (ms) | p99 (ms) | Max abs. error | Response bytes |
|---|---|---:|---:|---:|---:|---:|
| Windows | Naive per candidate | 1,712.72 | 1,793.79 | 1,843.92 | 2.30e-6 | 13,121,661 |
| Windows | Block packed | 427.59 | 484.33 | 486.75 | 2.24e-6 | 3,280,904 |
| Windows | One-response packed | 195.17 | 203.93 | 232.23 | 1.50e-5 | 131,876 |
| WSL2 Linux | Naive per candidate | 607.57 | 635.38 | 637.49 | 1.00e-6 | 13,121,661 |
| WSL2 Linux | Block packed | 149.31 | 160.66 | 166.42 | 1.01e-6 | 3,280,904 |
| WSL2 Linux | One-response packed | 169.96 | 172.89 | 173.18 | 1.47e-5 | 131,876 |

Every method passed the reported numerical tolerance.  The one-response
kernel's WSL median was 0.871 times its Windows median (about 13% shorter), and
its p95 ratio was 0.848.  The packed-versus-naive server-median speedup was
8.78 times on Windows and 3.57 times under WSL.  The wire reduction remained
99.5 times in both environments.

The smaller WSL advantage relative to naive is informative rather than a
failure of portability.  TenSEAL's exposed SEAL save/load path uses
runtime-local temporary files.  Linux `/tmp` made ciphertext serialization and
loading much cheaper: for example, naive client decryption had medians of
1,116.65 ms on Windows and 60.43 ms under WSL.  Conversely, the one-response
kernel retained its 131,876-byte response and stable arithmetic result across
both systems.  The data therefore support functional/runtime portability but
must not be presented as an independent-machine replication or a controlled
bare-metal OS comparison.

```text

  sha256 fc7b73d02efb0e18c000e9ad8b83480dfcd5dfd11065997ed4c6747470ae8915
tenseal-0.3.16-cp310-cp310-manylinux_2_27_x86_64.manylinux_2_28_x86_64.whl
  sha256 7add306c52ba1a2c7cd8307601fb8e1d04c520c9caec998a34d5bb6d4c5e1bcf
```

Full artifacts are
`results/system_revision/systems_expansion/windows_pair_ckks_micro.json`,
`wsl_linux_ckks_micro.json`, and the independently generated
`windows_wsl_runtime_pair.json`/`.csv`.  The comparison validator reports
`valid=true` and no errors.  The WSL setup and run commands are retained under
`system/wsl/`.

## Tests and interpretation limits

Run the harness-specific tests with:

```powershell
python `
  -m pytest tests\test_systems_expansion.py -q
```

They exercise exact frame accounting, key-material inspection, two concurrent
spawned workers, a real TCP connection, and a real TLS 1.3 connection.  The
current result is `3 passed`.

The harness and runtime-pair tests were also run together with the existing
packed-kernel and spawned-server tests; the combined result was `13 passed`.
Validate the
saved full artifact independently with:

```powershell
python `
  -m system.validate_systems_expansion `
  --input results\system_revision\systems_expansion\windows_full.json `
  --output results\system_revision\systems_expansion\windows_full_validation.json
```

The saved validation report is `valid=true`: it checks 19 conditions, 1,932
attempted requests, zero failures, 111 public-server role attestations, exact
key-component sums, amortization formulas, loopback/TLS status, and fixed
thread variables.  The full JSON SHA-256 is
`640b9098d310792c3b6ba52a4cd9f641614663bb55babebfc34e318563d249f9`.

The scaling workload uses deterministic synthetic normalized queries and
candidate lists against the actual million-vector provider memmap.  It is a
service-capacity experiment; BEIR relevance and candidate recall remain in
the separate graded-IR experiments.  The present evidence is still limited to
one Windows host, one CPU, and one HE library.  It does not measure a physical
network, multi-node scaling, or CKKS on the GPU.
